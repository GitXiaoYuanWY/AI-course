🔗 一、链接应用之API

我们平时使用的所有互联网应用——微信、支付宝、网易云音乐、ChatGPT……背后都在使用API。API（Application Programming Interface）是一种让不同软件"对话"的方式。

（一）API的原理

1. 通过一个简单的例子了解API

日常生活中寄快递的例子：

角色 对应关系 说明
寄件人（用户） Client / 用户 想寄一个包裹，但不自己跑到目的地。
快递单（API文档） API Documentation 告诉寄件人需要填写哪些信息（收件人、地址、物品类型等）。
快递员（API接口） API 接收包裹（请求），转交给快递中心，并把投递结果（响应）反馈回来。
快递中心（后台系统） Backend / Server 负责分拣、运输、投递等具体工作。寄件人不需要知道内部细节。

2. API的定义

API是一套标准化的规则，或者说是一个"密码"，支持一个软件程序与另一个软件程序之间的相互通信、交换数据和共享。

3. 总结

  • API：是用户应用与后台系统之间的桥梁，它允许我们调用和获取后台程序或系统暴露给我们的功能和数据。
  • API流程：
    • 用户应用：根据 API文档，通过 API发送 请求 (Request)。
    • API：将请求传达给后台系统。
    • 后台系统：处理数据，然后返回 响应 (Response) 给用户应用。

🌐 二、HTTP请求

（一）HTTP 请求是什么

1. HTTP请求的定义

客户端向服务器发送的，用于索取信息或执行操作的一行指令。

2. HTTP请求的工作流程

以日常浏览网页举例

  • 浏览器创建HTTP GET请求：当你在地址栏输入URL或点击链接时，浏览器首先会构建一个请求消息。
  • 请求发送至服务器：这个HTTP GET请求通过互联网传输到目标网站的服务器。
  • 服务器处理并返回响应：服务器接收并处理请求，然后返回包含HTML、CSS和JavaScript等内容的HTTP响应。
  • 浏览器渲染网页内容：浏览器接收响应数据，解析并将其转化为可视化网页界面。

用图书馆借书举例

  • 创建 HTTP GET 请求：走进图书馆，向图书管理员说明："我想借《瓦尔登湖》这本书。"
  • 请求发送至服务器：图书管理员收到你的借书请求后，开始在图书馆的数据库或书架系统中查找这本书。
  • 服务器处理并返回响应：图书管理员找到了这本书的资料，然后把结果交给你，告诉你可以借/已经被借完了。

（二）HTTP请求节点的组成

1. URL

URL的定义：

  • 互联网上资源的唯一地址，俗称"网址"。
  • 定义了要访问的资源位置，沿用快递的例子，他就是取件码，告诉你要在哪个货架上上面取货。

URL的组成部分：

  • 协议 (Scheme)：通常是 HTTP 或 HTTPS。
  • 主机 (Host)：你请求的后台系统的域名或 IP 地址。
  • 端口 (Port)：可选参数，用于指定连接端口。
  • 路径 (Path)：资源在服务器上的具体位置。
  • 查询参数 (Query Parameters)：以键值对的形式向服务器传递的参数，用于查询或过滤资源，例如 ?id=123&lang=zh。

注意：这些URL信息通常会在你想要请求的服务官方API文档中详细记录，可以直接复制使用。

2. Method（请求方法）

GET

获取数据，通常用于查询资源，不会改变服务器上的数据。

举例：就像查询图书馆中是否有某本书。

POST

提交数据，通常用于在服务器上创建新资源。

举例：类似向图书馆提交一本新书。

PUT

更新数据，完全覆盖现有资源。

举例：相当于替换图书馆中的某本书，用新版本完全取代旧版本。

DELETE

删除数据，从服务器中移除指定资源。

举例：就像将图书馆中的某本书彻底下架并从系统中删除记录。

HEAD

只获取响应头信息，不返回实际的响应体。

举例：类似于只查看书的封面和目录。

OPTIONS

查询资源支持的HTTP方法，用于检查资源是否存在/能否被访问。

举例：类似于询问图书管理员："我可以借阅、预订还是只能在馆内阅读这本书？"

PATCH

部分更新数据，只修改资源的部分属性。

举例：就像在图书馆的记录中，只修改书籍的一个字段，比如更新书的作者信息或分类标签。

3. Authentication（凭证）

根据凭据验明的访问者身份，一般就是输入API。

4. Query Parameters（查询参数）

以键值对形式传递的参数，用于查询或过滤资源。

5. Header (请求头)

提供了更多关于请求的信息，例如请求的地点、语言、数据类型等。

常见Header示例：

  • Accept: application/json：告诉服务器，我们期望返回的数据格式是 JSON。
  • Authorization: Bearer YOUR_API_TOKEN：用于身份验证，通常需要填入自己的 API Token 或 API Key。

6. Body (请求体)

包含了我们要发送给服务器的具体信息或数据。

示例：当调用 OpenAI 的图像生成 API 时，Body 中需要定义以下参数来生成图像：

  • "model": "dall-e-2" 或 "dall-e-3"：指定图像生成模型。
  • "prompt": "A cute baby sea otter"：图像生成的提示词 (Prompt)。
  • "n": 1：生成的图片数量。
  • "size": "1024x1024"：图像的分辨率。

注意：Body 的内容需要根据 API 文档内的具体要求和自身需求来填写。Body内容必须是合法JSON，否则请求会失败。

（三）实操案例：配置一个来自于外部模型的HTTP请求步骤

1. 打开DeepSeek的API文档

打开deepseek的API文档：https://api-docs.deepseek.com/zh-cn/

在首次调用界面，会有deepseek的详细的调用配置说明。

2. 找到DeepSeek的curl示例

在API文档中找到curl示例代码： 

curl https://api.deepseek.com/chat/completions \
-H "Content-Type: application/json" \
-H "Authorization: Bearer ${DEEPSEEK_API_KEY}" \
-d '{
     "model": "deepseek-chat",
     "messages": [
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "Hello!"}
      ],
      "stream": false
     }'

3. 开始配置HTTP节点

在N8N中选择 HTTP Request 节点

  • 将刚刚的 curl 整个复制
  • 点击 Import，n8n 会自动解析 cURL 命令
  • 自动填充 Method、URL、Header 和 Body 等所有配置项

4. 注意事项

  • 导入后，仍需手动将 Authorization Header 中的 YOUR_OPENAI_API_KEY 替换为自己真实的 OpenAI API Key。
  • 有时 n8n 自动解析的 Body 可能会将 JSON 结构拆分成多行参数，如果遇到问题，可以切换 Body 类型为 Using JSON，然后将 Body 的 JSON 内容整体复制粘贴进去。

（四）Webhook节点（API）

除了主动发起 HTTP Request，n8n 还提供了 Webhook 节点，它相当于一个反向的 HTTP Request。

1. Webhook节点的概念

HTTP Request 是我们主动向外部服务请求或发送信息，而 Webhook 则是 n8n 在等待外部服务或程序向我们发送信息。

2. Webhook节点的工作原理

1

Webhook 节点会生成一个独特的 URL。

2

一旦有 HTTP Request 发送到这个 Webhook URL。

3

n8n 的整个工作流就会被触发。

4

工作流会根据接收到的信息执行相应的处理。

5

最后，n8n 可以将处理结果作为 HTTP Response 返回给发起请求的外部服务或应用。

Webhook vs HTTP Request 对比

HTTP Request（主动）

我们主动向外部服务发送请求获取数据

Webhook（被动）

等待外部服务向我们发送数据触发工作流

3. 输入节点包名

在弹出的窗口中输入节点包名 → 点击安装

4. 使用节点

安装后，这个节点会出现在 "节点面板（Nodes）" 的列表中，像内置节点一样使用
